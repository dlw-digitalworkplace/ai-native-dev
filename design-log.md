# AI-native dev flow — Design Log
Last updated: 2026-06-25
Status: **Plan phase, build phase, and dreaming phase functionally designed** (D1–D18), plus a one-time **project onboarding** step that bootstraps the config layer (D18); **all open questions resolved or descoped** (one dreaming sub-item — human-decider emission — deliberately deferred). **Near-term scope: manual execution only** (D6 amendment, 2026-06-24) — unattended automation and the service-identity question are descoped for now. Remaining work is in the parking lot (deployment, etc.) and the deferred-until-automation items, not gaps in the design.

> Decisions are ordered by where they sit in the dev flow (overall shape → intake → status → planning → execution), not by when they were made.
> The **how-it-works** documentation lives in `design-doc.md`; this log holds decisions, rationale, and open items only.

## Decisions made

| # | Area | Decision | Rationale | Date |

| D1 | Overall flow + loop-back | **Intake (story gate) runs FIRST; human plan review (plan gate) runs LAST.** Rejections route by **root cause**, not always back to the story: bad story → edit & re-intake; plan-level gap → planner revises *in the PR*; story problem found at plan review → back to intake **and close the stale plan PR**. | The intake gate must be unskippable and sit before any planning cost. Routing by root cause avoids over-rotating to the story for every plan failure. | 2026-06-17 |
| D2 | Intake (phase 0) | An **intake agent** scores each user story *before* planning against a readiness rubric, writes its reasoning in ADO comments. It **suggests** fixes but does **not author** story content — the human authors the story/AC. | Catches vague stories before they burn downstream agent cycles; avoids polished-but-wrong AI-authored requirements. | 2026-06-17 |
| D3 | Status vehicle (cross-cutting) | Express status via **ADO tags** for v1 (not a custom field or the State machine), namespaced under `AIND` (AI Native Dev) — see D4. | Custom-field/process customization may be unavailable to the team; tags need zero setup. Namespacing makes them filterable and collision-free. **Known limitation:** tags are additive and uncontrolled — handled by the single-tag rule in D4. | 2026-06-17 |
| D4 | Status scheme (cross-cutting) | A single **`AIND status - <state>`** tag tracks the coarse phase. **Exactly one AIND-status tag per item at any time; every transition = remove-old-then-add-new, atomic.** States: `Ready for intake` / `Intake declined` / `Intake approved` / `Generating plan` / `Plan ready for review` / `Ready for implementation` / `In implementation` / `Implementation complete`. ADO tag = coarse phase; each GitHub PR (plan PR, then code PR) owns its fine-grained iteration (no PR-review states mirrored into tags). `Generating plan` and `In implementation` are kept as progress/diagnostic flags with a staleness timeout. Human-set: `Ready for intake`, `Ready for implementation`; agent-set: intake/plan states plus `In implementation`; set on merge: `Implementation complete` (mechanism resolved in D13). | Namespaced, queryable, respects request/assert. Coarse/fine split prevents two drifting sources of truth. The diagnostic in-progress flag aids trigger debugging (Q2). | 2026-06-17 |
| D5 | Planning / plan review | The implementation plan is reviewed as a **GitHub PR** (markdown in-repo), not an ADO attachment/comment. The PR is the review surface — inline comments, request-changes, revision diffs. **Refinement (2026-06-24): the planner's assumptions and open questions are recorded in two places and gate the merge.** Each assumption/open question goes (a) under the **Assumptions & open questions** heading in the plan markdown (self-contained, survives in the merged living doc) *and* (b) as an **individual resolvable PR review thread**. With the plan PR's target branch set to **require conversation resolution before merging** (a one-time repo-setting prerequisite), every assumption/open question **must be explicitly resolved by the human before the plan can be merged**. The mechanism is specifically *review threads* — plain top-level PR/issue comments carry no resolve state and would not gate merge. | PR review UX is built for iterating on a structured doc; ADO comments/attachments are not. **On the refinement:** putting each assumption in its own resolvable thread converts it from passive documentation into an active per-item checklist the reviewer must clear — a silent merge is, by construction, impossible while any thread is open, so the human cannot accidentally wave through an unexamined assumption. Keeping the heading in the doc too means the merged plan stays self-contained as living documentation (D10) rather than forcing a reader to reconstruct the assumptions from resolved threads. The branch-protection rule is called out as a setup prerequisite because the agent only *posts* the threads; the *gate* is enforced by the repo setting, not the agent — and it works under current manual scope because branch protection blocks a human's manual merge button just as it would an automated one. | 2026-06-17 (refined 2026-06-24) |
| D6 | Triggers / execution (cross-cutting) | **Execution matures in stages, all sharing one `.claude` config and the same agent definitions — only the trigger and identity change.** **v0 (local):** run each handoff from the Claude Code CLI (`/intake`, `/plan`, or `claude -p`) on a developer machine, acting as the developer's own ADO/GitHub credentials — no service identity, no CI — to prove the agents before any infra; plan-review revisions are re-run by hand. **v1 (GitHub Actions):** agents run via the official `claude-code-action`, kicked off by `workflow_dispatch` (human passes the work-item ID); plan-review revisions are event-driven via PR / `@claude` events (status unchanged); agents read/write ADO via a service identity; no service hooks, orchestrator, or staleness sweep. **v2 (pre-unlocked — service-hook admin confirmed):** swap the manual `workflow_dispatch` for an ADO **service hook → GitHub `repository_dispatch`** keyed on tag transitions (`Ready for intake` → intake; `Intake approved` → planner), plus a scheduled staleness sweep — same workflows, tag-driven. **The build phase (D7) extends the same model unchanged:** v1 `workflow_dispatch` kicks the coder (implement → polish → open PR) and the reviewer is PR-event-driven exactly like plan review; v2 keys the coder off the `Ready for implementation` → build transition. **AMENDMENT (2026-06-24): near-term scope is manual execution only.** The staged target above stands as the intended trajectory, but the *current* scope is deliberately narrowed to **manually-run execution** — i.e. v0 (local CLI, developer's own credentials) is the working mode for now. **v1's unattended-automation aspects are descoped:** the service-identity decision (single bot vs. per-agent, PAT vs. Entra SP, secret storage) is explicitly **out of current scope** and the question that covered it (formerly Q7) is **closed by descoping** — to be reopened when GitHub Actions automation is actually picked up. v2 (event-driven service hooks) is likewise future work. Nothing about the agent definitions, the flow, or the state model changes; only the automation/identity layer is shelved. | Local CLI is the lowest-setup way to validate the agents first; human-triggered GitHub Actions is the cheapest shared path (zero bridge infra), and the genuinely GitHub-native part — plan-PR iteration — gets real automation for free. Because every mode shares one config, promotion is zero change to agent logic; only trigger + identity move. Tags stay the state record throughout (D4), so v2 keys off them with no rework. Claude Code is first-party on GitHub Actions; ADO has no first-party task. **On the amendment:** scoping to manual execution removes the only thing that needed a service identity (an unattended agent), so the identity/secrets question stops being a prerequisite and can be parked without blocking progress — proving the agents by hand is the priority, and the automation layer is a clean later addition that touches trigger + identity only. | 2026-06-17 (amended 2026-06-24) |
| D7 | Build phase (coder → review) | **Once the plan PR merges, the build phase runs entirely in a code PR.** A **coding agent** implements the spec (`In implementation`); a **polish agent** (warm — shares the coder's context) does in-context cleanup (style, formatting, self-consistency) — not an independent check, by design. The coder opens the **code PR**; **CI** owns the objective gates. A **reviewer agent** — *cold: separate invocation, re-grounded only from PR diff + spec + project rules, never the coder's transcript* — reviews spec alignment + missed edge cases. Coder↔reviewer iterate in the PR for **≤3 reviewer passes** (a pass = reviewer comments → coder fixes or rebuts in-thread); the loop **exits early on reviewer approval**. Unresolved after the 3rd pass → **a human reads the threads and posts a verdict; the coder executes it; one final reviewer pass; then merge**. None of this loop moves the tag (coarse/fine split, D4) — the item stays `In implementation` until merge → `Implementation complete`. Coder and reviewer are each modelled as a single agent here, but either may be split into skill-specific agents per project. | Mirrors the plan-phase pattern: fine iteration in the PR, coarse state in ADO. The polish/reviewer split avoids paying twice for one checklist — warm polish does what the coder's context makes cheap (style, consistency); cold review is the genuine independent gate (spec, edge cases). Independence is enforced structurally, not by instruction. The cap with thread-visible disagreement hands the human the real exchange, not a synthesized summary; human-as-tiebreaker (verdict, coder executes) keeps the "agents do the work" pattern intact. | 2026-06-18 |
| D8 | Testing — TDD (point 1 of 2) | **Optional test-first step in the build phase, sitting between `Ready for implementation` and the coder.** **Trigger:** the **planner decides** whether a story warrants spec-level tests and records it in the plan; the human ratifies it at plan review (no new tag — rides the existing human gate). **Who writes what:** a **test-writer agent** — *cold, independent, same re-grounding contract as the reviewer (D7): re-grounded only from the merged spec + plan, never the coder's context, so it cannot be the same running session as the coder* — writes **failing behavioral / acceptance-level tests** (input→expected-behavior, tied to ACs, structure-agnostic). It does **not** write unit tests. The **coding agent** writes **unit-level tests** (warm) for the seams it creates, during implementation. **Review surface:** the behavioral suite is reviewed by a human **before implementation starts**, *inside* the implementation phase — **no separate PR and no new status** (coarse/fine split, D4: the fine-grained test review lives in the phase, the tag stays `In implementation`). **Reviewer's job shifts but does not shrink:** where a behavioral suite exists, spec-conformance is partly enforced mechanically (red→green), so the **reviewer (D7) refocuses on what tests can't express** — coding style, auth, logging, and other cross-cutting/project-rule concerns — plus the spec edge cases the suite missed; when no suite was written, the reviewer remains the sole spec-correctness check. **CI:** the test gate (previously empty) now runs the behavioral suite + the coder's unit tests. **Point 2 (live/E2E testing) is handled separately in D9.** | TDD's value is an *independent* executable encoding of the spec; a coder writing its own spec-tests is a mirror, not a gate (same failure mode as a warm reviewer), so the spec-test author must be cold. Behavioral (not unit) level is the test-writer's natural altitude from a *frozen spec*: unit tests presuppose implementation seams that don't exist pre-code, so a cold agent writing them would secretly design the code, constraining the coder in ways the human never reviewed — and you can't meaningfully review unit tests for code that doesn't exist (the reason the pre-impl review choice forces behavioral level). The hybrid (cold behavioral + warm units) puts independence where it counts and warmth where it's cheap, and gives full coverage overall. Keeping it optional avoids spending an agent on trivial stories; planner-decides-human-ratifies reuses the existing judgment gate instead of a lone machine tag. This finally puts something behind CI's test gate, so the cold reviewer is no longer the *only* spec-correctness check (the gap flagged in D7 / parking lot). | 2026-06-24 |
| D9 | Testing — live / E2E (point 2 of 2) | **Optional, pre-merge, blocking live/end-to-end gate in the build phase, on the code PR.** **Where:** pre-merge and blocking — runs on the code PR alongside CI and the reviewer; a failure blocks merge and routes back to the coder like any other failing gate. Stays at the same altitude as the rest of the build gates, so the terminal state is unchanged (`Implementation complete`) and nothing spills into the (still out-of-scope) deployment phase. **Trigger:** **planner-gated, exactly like D8** — the plan recommends whether the story warrants live testing; the human ratifies at plan review. A story now carries **two orthogonal test recommendations** (spec-level D8, live D9): either, both, or neither. **Who:** a **cold E2E agent** — same re-grounding contract as the test-writer (D8) and reviewer (D7): separate invocation, artifacts only, never the coder's context — authors and runs **user-visible behavioral scenarios** against the ACs (the full-stack altitude of D8's pattern). **Project escape hatches (by design):** the *how-to-run-the-app* mechanics are left to each repo because they are inherently per-app. A project may (a) implement automated E2E the cold agent drives, or (b) **skip the agent and have a developer run the app locally and test manually**, signalling the pass in the PR like any other human input. The design mandates *that the gate is cleared when the plan calls for it*, not how. **Status & CI:** no new status — the gate lives inside `In implementation` (coarse/fine split, D4), and a manual pass is just a human PR signal (no new machinery). The gate is "live testing happened," **not** "the CI E2E job is green": on the manual path there is no CI E2E job at all. | A blocking gate is the only version that is a real gate; advisory E2E that is often red gets ignored. Post-merge E2E would push the first live check past the terminal state into deployment territory that isn't designed yet, and would need a revert story — pre-merge keeps the gate honest and self-contained. Forcing a *running app* into every PR is exactly the per-app complexity worth refusing to pin down centrally, so the two escape hatches (automate it, or a human runs it locally) let each project choose its cost. Cold + behavioral + planner-gated keeps point 1 and point 2 a single coherent pattern — cold behavioral testing at two altitudes (unit-ish in D8, full-stack here) — rather than two unrelated bolt-ons. Signalling a manual pass in the PR reuses the existing human-in-PR pattern instead of inventing a manual-test checkpoint. | 2026-06-24 |
| D10 | Plan PR shape | **Separate plan-PR, merged to the project's integration branch first.** The plan is its own PR on its own branch and **merges to whatever branch the project treats as its integration/working trunk** (e.g. `main`, `develop`, or a `release/x.y` branch — the branch name is deliberately left to each project's branching strategy) **before any code branch exists**; the code PR comes later on a separate branch and targets the same integration branch. **Not** same-branch (plan + code co-evolving on one branch/PR). The plan markdown lives at **`/plans/<work-item-id>/plan.md`**, and it is **permanent living documentation** — it stays in the repo after the code ships, not a throwaway. This makes "the merged spec" that D7/D8/D9's cold agents re-ground from a concrete, stable, addressable artifact. | The entire build phase rests on a **frozen spec**: the cold reviewer (D7), cold test-writer (D8), and cold E2E agent (D9) all re-ground "from the merged spec," which only means something if the spec is genuinely merged and immutable before build begins. Same-branch would make the spec a moving target on the branch and turn the plan gate soft. The only cost — a second PR per story — is cheap next to the independence guarantee it buys. The branch name is left open because branching strategy is per-project (trunk-based, git-flow, release branches) and pinning it centrally would impose a workflow we don't own; the design only requires that the plan merges to the integration branch *first*. A stable `/plans/<id>/plan.md` path gives every cold agent a deterministic re-grounding source; keeping it as living docs means the spec stays discoverable next to the code it produced. | 2026-06-24 |
| D11 | Readiness rubric (intake / phase 0) | **The intake agent (D2) scores stories against a two-layer, hybrid rubric stored at `.claude/intake-rubric.md`.** **Hybrid structure:** *objective* criteria are pass/fail — any miss → `Intake declined`; *judgment* criteria are raised by the agent as advisory comments on the work item, not hard fails (matching D2: the agent suggests, it does not author). **Two layers, single seeded file:** the flow ships a starter `.claude/intake-rubric.md` containing the baked-in **core**; each project edits that file in place to extend it, adding both objective *and* judgment criteria for project-specific needs (e.g. "touches payments → must cite PCI checklist"). **Core is a strong default, NOT an enforced floor** — because it is a single editable file (not a read-only merged layer), a team *can* delete core items; nothing structurally prevents it. **Baked-in core — objective (pass/fail):** title present and non-trivial; ≥1 acceptance criterion exists; intent is stated (user-story form *or* a clear problem/goal statement — "As a…" not forced rigidly); no unresolved placeholders (`TODO`/`???`/`TBD`) in body or ACs; dependencies/blockers named or explicitly "none". **Baked-in core — judgment (advisory comments):** ACs are testable/observable, not subjective ("works well"); story is single-sized, not an epic in disguise; story is internally coherent (title, intent, ACs agree); enough context for a planner to act on the *what* without guessing (the *how* is the planner's job). | Intake is the first and only unskippable gate (D1); without explicit criteria, `approved`/`declined` is noise and the human re-litigates every call, defeating the gate. Hybrid puts determinism where auditability matters (objective checklist) and judgment where a checkbox can't catch the real failure — a well-formatted but meaningless AC. Two-layer mirrors the existing shared-config-plus-project-rules pattern: a core so no project ships with *no* rubric, an extension so teams encode their own context. Single seeded file (Option A) chosen over an enforced read-only core (Option B) for simplicity and to trust teams — the accepted cost is that the core is a default, not an invariant; if enforcement is later needed, promote the core to a read-only merged layer. The rubric lives in-repo so the intake workflow checks it out with the rest of `.claude`, and so changing scoring criteria never touches agent logic. | 2026-06-24 |
| D12 | Exception handling — stuck-state protocol (cross-cutting) | **One generalized stuck-state protocol covers every "agent cannot make progress" failure: planner can't produce a viable plan after N tries, coder can't get the behavioral suite green after N tries, and a merge conflict from concurrently landed work.** Mechanism (identical for all three): each retry loop has a fixed **attempt cap** (the actual N is a tuning detail, parked — set the mechanism now, the numbers empirically later); on hitting the cap the agent **stops** — no infinite retry, no merge, no silent exit. It sets a single shared **`AIND status - Needs attention`** tag (replacing the current status as one atomic change, per D4) and posts **what it tried and why it is stuck** as a work-item comment (planner / coder) or PR comment (coder-in-PR), so the human sees the real trail, not a synthesized summary (same principle as D7). **Merge-conflict special case:** the coder first attempts **one automated rebase/conflict-resolve**; it only escalates to `Needs attention` if that single attempt fails. **Resume:** a human resolves the blocker, then **re-triggers the phase the item was stuck in** — `Needs attention` returns to its origin phase (planner-stuck → re-run planning; coder/merge-stuck → resume build). In **v1 the human picks the right workflow to re-dispatch** (they read the comment and know whether to re-run plan or build); **v2's automatic re-routing from `Needs attention` is deferred** along with the rest of the v2 trigger work. **`Needs attention` is distinct from D7's human-tiebreaker:** D7 (agents *disagree* but are still working) deliberately does NOT move the tag — the item stays `In implementation` and the dispute lives in the PR; `Needs attention` is for an agent that has *stopped and cannot proceed*. | Every agent is triggered and headless — nothing watches it run — so each failure mode needs an explicit "stop and flag a human" exit, or a stuck agent loops forever burning tokens or fails silently while the item sits looking active. Generalizing D7's already-proven shape (cap → stop → escalate with visible context) to one protocol avoids three bespoke mechanisms for one problem. One shared status keeps the coarse/fine split honest (D4): the finer "which failure" is recoverable from the comment, so it doesn't need its own tag and doesn't bloat the state machine. Parking N avoids guessing empirical thresholds pre-implementation. The single automated rebase attempt handles the common trivial-conflict case cheaply without a human, while still escalating the genuine conflicts that need judgment. Returning to the origin phase (rather than a fixed restart point) is correct because the three cases are stuck at different points; v1 leans on the human-as-watchdog already assumed for triggering (D6), so no new routing machinery is needed until v2. Keeping it separate from D7 preserves the meaning of each: "we disagree, keep going" vs. "I am stuck, stop." | 2026-06-24 |
| D13 | Merge mechanics (build phase) | **In current (manual) scope, the merge is human-gated and a single CLI command performs the merge plus the terminal tag-write together.** **Merge trigger:** a person merges the code PR after the reviewer agent approves (or after the D7 human-verdict + final reviewer pass). This is committed as the design position for manual mode, not a stopgap — a human is the final authority on what lands. **Tag write:** the terminal `Implementation complete` tag is written by a **CLI command that also performs the merge** (folded into `/implement`'s final step, or a dedicated `/complete <work-item-id>`), acting as the **developer's own credentials** (the v0 model, D6) — no service identity. **Order and failure handling: merge FIRST, then write the tag.** If the tag-write fails after a successful merge, the item is left **recoverable** — it keeps `In implementation` and the human re-runs the tag step — and is *never* falsely marked complete; the reverse order is forbidden because a `Implementation complete` tag on an unmerged PR would make the state record lie (the discipline behind D4's single-tag invariant). **Explicitly deferred to the automation phase:** auto-merge-on-approval, and a PR-merge GitHub Action that writes the tag as the service identity — both presuppose the unattended machinery and service identity descoped under the D6 amendment (former Q7), so they wait until automation is picked up. | Manual scope (D6 amendment) removes the service identity that the original Q8 assumed, so the realistic answers collapse to human-gated + CLI. Doing merge and tag-write in one command keeps them from drifting — a merged PR with a stale `In implementation` tag is exactly the two-sources-of-truth failure D4 exists to prevent. Merge-first ordering makes the only possible failure a *recoverable, visibly-incomplete* item rather than a *false-complete* one: a lie in the state record is far worse than a missed update, since downstream readers trust the tag. Committing to human-gated (rather than holding it loosely) reflects that even once automation returns, a human merge gate is a defensible permanent choice; auto-merge is offered as a *future option*, not a planned default. Deferring the automation-mode answer keeps Q8 consistent with the Q7 descope instead of designing machinery that isn't in scope. | 2026-06-24 |
| D14 | Behavioral suite shape (build phase) | **The cold test-writer's behavioral suite lives on the SAME branch as the implementation (one PR carries red→green), reviewed in place before the coder writes code — NOT a separate test-PR merged first.** The test-writer (D8) commits the failing behavioral suite as the first commit(s) on the code branch; a human reviews the suite **at the test-only state of the branch** — after the test-writer's commits, before any implementation commits are pushed (per D8: reviewed before the coder starts, inside the implementation phase, no new status). The coder then adds implementation to the same branch, turning the suite green, and the single code PR carries the whole red→green history. The exact PR-opening timing (review the branch at the test-only commit, or open the PR early as tests-only and review its first state) is a mechanical detail left open; it does not change the design. **Guard:** because the suite is *not* merge-frozen (it sits on the branch, where the coder could in principle weaken it), the **cold reviewer (D7/D8) treats any unexplained change to the behavioral test files as a review flag** — re-grounded from the spec, it checks the tests still match the ACs. This closes the "coder weakened the tests" gap that a merged-first test-PR would have closed by freezing. **This deliberately diverges from D10** (which chose separate-PR-merged-first for the *plan*). | The artifact coupling is the opposite of the plan case, so the same-branch-vs-separate-PR answer flips. The plan is *loosely* coupled to code, so D10's merge-freezing adds integrity at low cost. The behavioral suite is *tightly* coupled (same language/repo, designed to go red→green against the code), so separate-PR has high cost and low benefit here: (1) D8's independence is already enforced **structurally** (cold invocation, separate context) — it never depended on branch isolation, so freezing the suite on trunk adds little integrity; (2) merging a **red** suite to the integration branch first would break trunk CI for everyone branching off it, violating the basic "trunk is green" invariant — a real harm, not theoretical; (3) tightly-coupled artifacts want to travel and merge together, so splitting them is the unnatural move (the inverse of the plan, where splitting was natural). The reviewer-flags-test-changes guard recovers the only integrity that same-branch gives up, and it's catchable rather than silent because test-file edits appear in the same PR diff the reviewer already reads. Net: principled divergence from D10, not inheritance. | 2026-06-24 |
| D16 | Dreaming phase — continuous improvement loop (cross-cutting, non-linear) | **A non-linear, cross-story phase that learns from the flow's exhaust and proposes improvements to the agent-config layer.** Four parts. **(1) Emission (push, warm):** each agent emits a structured *lessons-learned* record at the end of its session — e.g. the planner after iterating on plan-PR comments, the coder/polish after a recurring style fix. The agent is the correct *warm* source because only it knows what it tried and why it iterated; this self-report is deliberately warm, exactly the kind of self-grading ruled out for *gating* (polish D7, test-writer D8) but appropriate for *emission*. **(2) Synthesis (cold dreamer):** a separate **dreamer agent** — *cold, same re-grounding contract as the reviewer (D7) / test-writer (D8) / E2E agent (D9): a fresh invocation re-grounded only from the emitted lessons + artifacts, never any agent's running context* — reviews the accumulated lessons and proposes improvements. The split is **reflect-warm / synthesize-cold**, mirroring D7/D8: warmth where it is cheap (an agent self-reporting its own session), independence where it counts (deciding what is a real, recurring pattern versus a one-off). **(3) Cadence:** runs on a regular basis; the framework leaves the exact trigger open — **e.g. every X `Implementation complete` items** (a volume-based example, not pinned, same spirit as parking N in D12). **(4) Output & authority:** every proposal lands as a **GitHub PR against `.claude`**, **one dream cycle = one PR**, which a **human accepts or rejects** — the dreamer *proposes*, the human *disposes*. This human gate is **permanent, not a manual-scope limitation** — the D2/D13 *agent-suggests-human-decides* pattern applied to the highest-blast-radius write in the system. **Authority is hard-bounded to the config / "learned behavior" layer** — skills, agent prompts, the intake rubric (D11), project rules — and **never the flow itself**: the status model (D4), the gates, and the structural decisions D1–D15 are off-limits. The dreamer may *raise* a suspected structural problem as a **parking-lot note for a human**, but must **not** propose structural changes as mergeable diffs. Improving *how* an agent does its job ≠ redesigning *the job*. **Deferred sub-item:** the human decider (D7 tiebreaker) and human merger (D13) are **not agents and do not auto-emit**, so the highest-value datum — *why a human overrode the machine* — is not captured by the emission step; capturing it is deferred (it would add friction to the human gate, and is a gate-mechanics detail, not framework shape). | The whole flow is a system of agents reading `.claude`; until now **nothing closed the loop on improving that config** — a linear pipeline with no feedback path. The dreamer adds that path, but it is structurally the **most dangerous component in the system**: it is the first phase that is asynchronous, cross-story, and **self-modifying** — it *writes the very files every other agent reads*, so a bad lesson silently degrades every future run rather than breaking one story. That blast radius is why it gets the **strictest** controls, not the loosest: the permanent human merge gate (it cannot change agent behavior without a human merging a PR), the cold-synthesis guarantee (warm self-grading must not decide what counts as a pattern), the one-PR-per-cycle shape (each change individually rejectable, auditable, and revertible — D5's resolvable-thread philosophy at the file level), and the hard authority boundary keeping it on the **config side of the D1–D15 line** (a self-improving agent that could rewrite its own gates is exactly the failure to design out). Emission is warm because the agent is the only thing that knows the *why* behind its iterations — artifacts show *that* the planner iterated three times, not *what it learned* — and the dreamer's coldness is precisely what keeps the synthesis of those warm reports honest. | 2026-06-25 |
| D15 | E2E execution environment (build phase) | **In current scope, the live/E2E gate (D9) is satisfied by the manual developer-run path only; the automated E2E path is deferred.** When a story's plan calls for a live test (D9), the gate is cleared by a **developer running the app locally and testing manually, signalling the pass in the PR** — exactly the manual escape hatch D9 already defines. The **automated path** (a cold E2E agent stands up the running app in CI and drives user-visible scenarios) is **not buildable in current scope** and is deferred: it depends on the unattended automation layer (descoped under the D6 amendment) *and* on a way to stand up a running application in a PR/CI context — ephemeral CI service, preview/dev deploy, or containerised run — which overlaps the **deployment phase that is in the parking lot and out of scope for the initial design**. **D9's two-path design stays intact as the target:** when both the automation layer and a deployment/preview story exist, the automated path becomes real and projects choose per-repo as D9 specifies. This decision only names what is actually available today (manual) versus deferred (automated) — it does not narrow D9. | Q10 is entirely about the *automated* path; the manual path needs none of this infrastructure and is already fully specified in D9, so it is the natural — and currently the only available — answer. Stating "manual now, automated deferred" (rather than leaving it nominally "per-project") is the truthful framing: a project *cannot* take the automated path right now because both prerequisites are deferred, so "your choice" would be a false choice in current scope. | D18 | Project onboarding (one-time, pre-flow; bootstraps the config layer) | **A human-invoked onboarding agent (`/onboard`) bootstraps a project's `.claude/` config from its existing codebase, as the day-one mirror of the dreamer (D16).** It reads the repo, **discovers the rule areas it actually needs through three lenses** — **technical layers** present (front-end, back-end, web-jobs, infra, …), **cross-cutting concerns** with a notable/non-standard approach (e.g. a custom pin-code auth scheme), and the **functional/domain architecture** (the product's core concepts and invariants, e.g. "app-of-mini-apps; every entity scoped to a couple of IDs") — and **drafts** the initial config: one `rules/<area>.md` per discovered area, a `CLAUDE.md` that imports exactly those files via the root-import model, **project-specific skills** stubbed from the real build/test/run commands it finds (manifests, CI), and a copy of the seed intake rubric (D11). **Refinement (2026-06-25):** the layer names are *examples, not a fixed list*, and discovery is **strictly evidence-only** — a rule file is created only for an area that genuinely exists (no test framework → no testing rule; no docs system → no docs rule), and the three-lens prompt was added so onboarding does not stop at technical layers but also surfaces cross-cutting concerns and the functional/domain architecture. The shipped `project-template/rules/` is a single category guide (`_TEMPLATE.md`), not a set of stub files, so nothing biases the agent toward a specific domain set. It then runs a **preflight probe** and reports the remaining setup prerequisites (PAT, gh access, jq, the Boards↔GitHub integration, branch protection). **Authority & gate:** it **suggests, the human decides** (the D2 pattern) — every generated file is a clearly-marked **DRAFT** for review, and it is **bounded to the config layer, never the flow** (the same D1–D15 boundary the dreamer respects, D16). **Delivery:** draft files written directly into `.claude/` (reversible via git, reviewed before commit) — **not** a PR-against-`.claude`, because the PR-review machinery (D16's one-cycle-one-PR) presupposes a running flow and a configured repo that do not yet exist at onboarding. | Adopting AIND on an existing repo otherwise starts from empty template stubs a team fills by hand; turning the codebase itself into the first draft of its rules is faster and more grounded. Reusing the agent-suggests/human-decides gate (D2) avoids inventing a new trust model, and keeping the onboarder on the config side of the D1–D15 line is the same failure-class avoidance as the dreamer — a setup agent that could rewrite the flow is exactly what that boundary designs out. Draft-files-not-PR is the right call *specifically at onboarding*: the human gate is preserved (review before commit) without the bootstrap chicken-and-egg of needing the flow configured to review its own configuration. Discovering domains rather than fixing them keeps the rule split honest to each repo's real shape, and the root-CLAUDE import model (chosen for project rules) wires in exactly the domains found. Folding the preflight probe into onboarding means the team finishes the command knowing precisely what remains to set up. The onboarder/dreamer symmetry is deliberate: **onboarding bootstraps the config from the codebase, dreaming evolves it from the flow's exhaust** — two ends of the same agent-config-improvement axis, both proposing, both human-gated, both barred from the flow. | 2026-06-25 |
| D17 | Artifact linking & resolution (cross-cutting) | **How an agent resolves between a work item, its plan, its branch, and its PRs is a defined contract — not left to a naming convention.** Two layers. **(1) Framework convention (the fast path):** the plan markdown lives at **`/plans/<work-item-id>/plan.md`** (D10), and the **work-item ID is the join value** carried in artifacts. **Branch naming is a project responsibility, not framework** — the framework never constructs, assumes, or recognises a branch name; reaching the branch goes *through* the code PR, not by deriving a branch string. **(2) Stored pointers (the source of truth):** the plan PR and the code PR are each **linked to the ADO work item via ADO's native work-item↔PR linking** (the Azure Boards ↔ GitHub integration). **In addition**, each PR body carries a fixed, machine-parseable `AIND-LINKS` block (an HTML comment, invisible in rendered PR text) listing the **work-item URL**, the **plan path**, and — on the code PR — the **plan-PR URL**. Both are kept on purpose: the native link is the canonical pointer and the human-visible one; the in-body block is what lets a **cold agent re-grounding from artifacts only** (the PR diff/body) resolve the work item and plan **without an ADO round-trip**, keeping "re-ground from artifacts" literally true. **Resolution rules:** *forward* (have ID → plan file by the `/plans/<id>/plan.md` convention; ID → PRs by following the native links from the work item); *reverse* (have a PR → work item and plan via the native link and the `AIND-LINKS` block). **Conflict rule:** if a convention-derived guess and a stored pointer disagree, the **stored pointer (native link) wins and the agent flags it** — this is the survived-a-rename / hand-edited case. **Prerequisite (named, like D5's branch-protection rule):** the **Azure Boards ↔ GitHub integration must be connected** for native links to exist; if it is not, the degraded fallback is a fixed-prefix work-item comment (`AIND-LINK: <kind> <url>`). Native is primary, the comment is the fallback. **Failure handling (mirrors D13's merge-then-tag):** the link/back-pointer is created **after** the PR is successfully opened, so a failed link leaves a real, convention-discoverable PR rather than a dangling pointer to nothing — recoverable, never corrupt. **Scope:** entirely v0-CLI-doable under the developer's own credentials; under automation the same links are created by the service identity, contract unchanged (D6). | The whole cold-agent design (D7/D8/D9/D16) rests on agents re-grounding "from the merged spec," but until now nothing said how an agent handed only a work-item ID *finds* the spec, the branch, or the PR — a naming convention was silently load-bearing, and a renamed branch or hand-edited title would break the join with no detection, and there was no reverse pointer at all. A two-layer contract fixes both directions cheaply: the convention stays the fast path, and a canonical stored pointer makes reverse lookups robust and rename-proof. Native ADO links were chosen over greppable comments because they are a built-in link type (not the field/process customization D3 warned might be unavailable), they are human-auditable from the Boards UI, and they need no parsing — with the integration-connected prerequisite called out explicitly so the dependency is visible, exactly as D5 surfaces its branch-protection prerequisite. The in-body `AIND-LINKS` block is kept *as well* because a cold agent's contract is "artifacts only": it sees the PR body but does not necessarily query ADO's link API, so the in-body block is what keeps that contract honest without an extra round-trip. Branch naming is left to the project for the same reason D10 leaves the integration-branch name open — branching strategy is per-project and pinning it would impose a workflow the framework does not own — which is *why* the branch is reached through the PR rather than by deriving a name. Stored-pointer-wins-and-flags makes a rename a *detected, recoverable* event rather than a silent break, and create-link-after-PR-open makes the only possible failure a recoverable-but-discoverable item, the same shape as D13's merge-first ordering. | 2026-06-25 |
